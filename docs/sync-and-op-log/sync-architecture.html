01 / five-minute route

Why the system has this shape

The operation log is the smallest shared mechanism that works offline, crosses multiple transports, and can reason about concurrent edits while the remote side remains unable—or untrusted—to merge user data.

  • Core six-step trace below
  • Deep expandable mechanics
  • Source executable or canonical owner

Five-minute spine: read only this six-step trace for the basic model. Stop after step 06, or follow its links into the broad tour and disclosures.

  1. 01 / device A

    Edit and capture

    A title edit reduces into live state immediately. Capture then appends one replayable operation and its vector clock to SUP_OPS.

  2. 02 / device B

    Edit independently

    Before seeing A's title edit, B completes the same task offline and captures its own operation. Neither device's clock includes the other edit.

  3. 03 / transport

    Download first

    A's next cycle downloads first, then uploads its title operation. B's later cycle downloads that unseen operation before uploading B's completion. SuperSync pages operations; file providers adapt shared envelopes.

  4. 04 / resolver

    Classify intent

    Before reducer apply, the incoming title is compared with B's local frontier. Vector clocks prove concurrency; safe disjoint fields merge before semantic precedence or deterministic LWW.

  5. 05 / durable settle

    Apply and converge

    The resolved work is staged, reduced, and durably checkpointed before the cursor advances. A synthetic winner covers both histories and re-uploads in the bounded current-cycle loop.

  6. 06 / restart

    Recover the projection

    Startup selects a usable cache, reconstructs its tail, and resumes pending checkpoints. The cache accelerates recovery; retained evidence makes it safe.

i

Authority boundary

This page owns the whole-system mental model. Exact contributor rules, clocks, conflict policy, package ownership, encryption behavior, wire formats, and schema compatibility remain owned by their linked documents and code.

Deep / why not synchronize whole state or use server timestamps?

Whole-state replacement cannot reconstruct independent offline intent, and wall clocks cannot prove causality. A trusted server-side merge is also unavailable to file providers and incompatible with opaque encrypted payloads. The rejected alternatives and their trade-offs live in Operation Log Architecture — “Why This Architecture: Rejected Alternatives”.

02 / facet: local persistence

Three representations, three jobs

“The state” is overloaded. The running projection, durable history, and restart baseline are related but not interchangeable; most recovery bugs begin by treating them as one thing.

Live / NgRx

Runtime projection

Reducers update the state the UI reads. It is authoritative for the running session, but memory alone is neither restart-safe nor syncable.

Durable / ops

Operation history

SUP_OPS.ops stores local and remote operations, source, synchronization markers, and crash-recovery application status.

Restart / state_cache

Replay accelerator

A structurally usable snapshot plus its last applied sequence and vector clock avoids replaying the entire retained log on every start.

Adjacent stores and remote copies
Projection / archives

Archive stores

Young and old archive object stores live inside SUP_OPS. Remote archive effects are deterministic, explicit, and retryable.

Remote / provider

Delivery substrate

SuperSync retains ordered operations, including causal full-state boundaries; file providers also carry snapshot baselines. Neither is the running UI state.

Dormant capability / journal

Conflict review store

SUP_CONFLICT_JOURNAL is separate, device-local, best effort, and never allowed to influence conflict outcomes. The main remote-conflict path currently disables emission, so it is not complete evidence.

!

This is not a conventional write-ahead log

The reducer runs first; operation capture and durable append follow asynchronously. A failed append can therefore leave live state ahead of disk. The pending-write counter and divergence guard suppress snapshotting until a reload can restore a trustworthy projection.

Deep / persistence and compaction safety
  1. Treat state_cache as a screened replay baseline, never as a second source of truth.
  2. Keep pending or archive-pending remote work durable until its reducer and side-effect checkpoints finish in order.
  3. Save a new snapshot only when state is valid and no persistence divergence or hydration fallback makes the projection suspect.
  4. Prune only old terminal operations already represented by the snapshot; retain the tail needed for conflict frontiers.

The guarded direct-load, migration, replay, and fallback cases are centralized in restart recovery.

Sources: database schema, hydrator, compaction, and conflict journal contract.

03 / device A

One replay-atomic transition becomes one operation

A persistent action is both the reducer instruction and the replay contract. Consequences that must replay atomically happen inside the same reducer pass. A broader UI workflow may deliberately compose independent actions when their normal side effects and conflict boundaries matter.

  1. 01 / dispatch

    Persistent action

    The UI expresses one replay-atomic transition with entity metadata.

  2. 02 / reduce

    State changes

    NgRx and task-shared meta-reducers run synchronously.

  3. 03 / count

    Capture pending

    The capture meta-reducer records work after reducers return.

  4. 04 / build

    Create operation

    The effect adds identity, timestamp, schema, and vector clock.

  5. 05 / commit

    Append + clock

    The operation and durable clock advance atomically in SUP_OPS.

  6. 06 / ready

    Pending upload

    The write counter clears; the operation is available to sync.

Atomic pattern

Reducer cascade → one operation

Deleting a tag and removing it from its tasks belongs in one meta-reducer pass. Replaying the original action reproduces the same complete transition.

Footgun

Effect fan-out → several operations

Follow-up dispatches used only to implement one replay-atomic transition introduce partial states and can re-fire during replay unless the action boundary is correctly filtered.

Deliberate workflow composition is different: independent actions may keep their normal local effects and entity-specific conflict boundaries. Project completion is the canonical example; see ADR #5: Project completion.

!

Reducers replay; ordinary effects do not

Ordinary remote and replayed operations pass through reducers to rebuild state; restart recovery owns the guarded shortcut and fallback rules. Effects inject LOCAL_ACTIONS so replay cannot repeat UI side effects or emit phantom operations. Selector-driven mutating effects additionally need the sync-window guard.

Deep / the contributor decision table
When adding… Required shape Why
An action-based effect Inject LOCAL_ACTIONS Remote and replayed actions have already produced their effects.
A selector-driven mutating effect Prefer an action; otherwise use skipDuringSyncWindow() Selector emissions bypass the local-action boundary.
One replay-atomic transition changing several entities Use a meta-reducer, not effect fan-out One reducer pass is captured as one replayable operation.
A loop dispatching many operations Yield once after the loop The asynchronous capture pipeline must observe each intermediate state.

Canonical contract: The Contributor Sync Model. Executable owners: capture meta-reducer, persistence effect, and LOCAL_ACTIONS.

Deep / local input that lands during remote apply

The reducer still accepts the user's action, but immediate operation capture is deferred. After remote clocks and reducers are durably settled, buffered actions are drained and receive clocks based on the new remote frontier. This prevents a genuine local edit from being dropped while avoiding an operation that is obsolete at birth.

04 / sync cycle

One orchestration contract, two transports

The app synchronizes through an operation-oriented provider port. SuperSync implements it directly; file providers are adapted to it. The ordinary cycle downloads and settles remote work before uploading local work.

Transport A / SuperSync

Ordered operation relay

PostgreSQL assigns a per-user server sequence for delivery and cursoring. Upload conflict checks compare causal metadata before storage. A causal full-state operation lets late clients fast-forward and authorizes pruning only the prefix before that boundary; without one, old history remains.

  • WebSocket messages trigger downloads; HTTP moves data.
  • Encrypted payloads remain opaque to the server.
  • The optional snapshot cache is not part of normal sync downloads; the server can use it to generate historical restore state returned to a client.
  • Server sequence establishes delivery order, not causality.
Transport B / file v2

Shared monolithic envelope

sync-data.json combines state, archives, vector clock, and a bounded recent-operation window. Dropbox, WebDAV/Nextcloud, OneDrive, and LocalFile share this adapter format, but not the same physical write guarantee.

  • Dropbox and OneDrive can enforce compare-and-swap through their provider APIs.
  • WebDAV/Nextcloud is atomic only with a strong ETag; weak or missing ETags fall back to a best-effort content check before PUT.
  • LocalFile uses a best-effort read/check/write sequence and is single-writer/backup-only.
Transport B / file v3 opt-in

Split logical commit point

sync-ops.json is the small logical commit point. Normal compaction can reference a generation snapshot containing state and archives, so ordinary op-only sync avoids rewriting the heavy snapshot. Explicit snapshot and recovery paths can still use fixed sync-state.json.

  • Compaction writes the snapshot before publishing its pointer.
  • A v3 tombstone prevents older v2 clients from diverging.
  • Backup files and migration markers make interrupted writes recoverable.
  • Pointer publication has the same backend-specific CAS limits as v2.
Shared client contract

Same incremental path; separate baselines

Both transports expose download, upload, snapshot, cursor, and reset operations through the same client port. Incremental operations share migration, conflict, and reducer replay. Fresh file bootstrap and gap recovery instead atomically install a snapshot baseline, reject superseded pending local ops, and deliberately create no SYNC_IMPORT.

↯

Do not collapse three different races into “conflict resolution”

  • On download, the client compares entity frontiers and resolves concurrent operations.
  • On SuperSync upload, the server can reject an operation and return the competing operation; the client then creates any dominating replacement.
  • At file commit, Dropbox/OneDrive and strong-ETag WebDAV/Nextcloud enforce CAS, so a revision mismatch aborts the envelope write and requires a fresh download before retry. Weak/missing-ETag WebDAV and LocalFile only perform best-effort checks that cannot close the read/write race.
!

The file adapter does not merge at the provider

The adapter reads a provider revision and builds the next envelope locally, but physical CAS depends on the backend. Dropbox/OneDrive APIs and strong-ETag WebDAV/Nextcloud can close the race; weak or missing ETags and LocalFile use a separate best-effort read/check/write sequence. LocalFile is therefore single-writer/backup-only. The envelope's syncVersion is not a provider revision or ETag.

Deep / transport owners and privacy boundary

Orchestration lives in SyncWrapperService. Provider adaptation and fencing live in WrappedProviderService. File envelopes and durable names live in @sp/sync-providers. The detailed ownership contract is Sync Package Boundaries. SuperSync's transport, retention, and trust boundaries are detailed in the SuperSync Server Architecture.

SuperSync can read routing and conflict metadata needed to operate the protocol. With E2EE enabled it cannot inspect operation or snapshot content; see SuperSync End-to-End Encryption Architecture.

File providers can optionally encrypt each whole file envelope. That choice is stored independently in each provider's private configuration and does not inherit SuperSync's encryption setting. See the focused provider-switch scenario I9 and its executable owner, WrappedProviderService.

05 / device B

Remote apply is a recovery state machine

Downloading bytes is not acknowledgment. An operation becomes cursor-safe only after compatibility, conflict policy, durable checkpoints, reducers, and required archive side effects succeed. Archive failure keeps the cursor fixed and leaves durable retry evidence.

  1. 01

    Migrate the incoming prefix

    Receiver-side migrations run before interpretation. A too-new operation blocks itself and the suffix so the cursor cannot skip unknown semantics.

    compatibility gate
  2. 02

    Apply full-state filtering rules

    Explicit imports discard invalidated history; causal repair uses a narrower policy. Ordinary operations continue to entity conflict detection.

    causal boundary
  3. 03

    Classify conflicts against local frontiers

    Pending writes are flushed first. Vector clocks identify causal successors, duplicates, stale operations, and genuine concurrency.

    decision
  4. 04

    Append accepted remote operations as pending

    Duplicate IDs are skipped atomically; every newly accepted row exists on disk before reducer application begins.

    durable
  5. 05

    Durably merge incoming clocks

    The receiving frontier advances before the reducer window, so deferred local actions can later receive clocks based on committed knowledge.

    durable
  6. 06

    Bulk-apply through reducers

    The original actions replay in order through the same reducers while ordinary local-only effects remain filtered out.

    projection
  7. 07

    Checkpoint reducer success

    Successful rows become archive_pending; reducer failures are quarantined. The checkpoint commits before archive work starts.

    durable
  8. 08

    Run deterministic archive side effects

    Completed rows become applied. An attempted failure becomes failed, unattempted successors remain archive_pending, and the apply call aborts for retry.

    terminal
  9. 09

    Advance the provider cursor

    The cursor advances only after the downloaded page applies successfully. A failure leaves it unchanged. A crash after apply but before this write causes a harmless redownload whose operation IDs deduplicate.

    acknowledge
Deep / what each crash window does on restart
Crash window Durable evidence Recovery
After pending append, before reducers pending operation row Startup replays reducers for the same row.
After reducer checkpoint, before archive completion archive_pending Startup reruns only the idempotent archive side effect.
During one archive side effect failed attempted row; archive_pending successors; unchanged cursor The cycle aborts; startup or a retry resumes ordered archive work.
After apply, before cursor write applied rows and operation IDs Redownload; append skips duplicates; cursor catches up.

Executable ordering: applyRemoteOperations(). Host orchestration: RemoteOpsProcessingService. Archive contract: ArchiveOperationHandler.

!

Never draw the cursor beside “download”

Advancing it before durable application turns a crash into permanent data loss: the provider believes the client has consumed operations that no local recovery row can replay.

06 / two devices, one conflict

Sequence orders delivery; clocks prove causality

A server sequence answers “what should this client download next?” A vector clock answers “had this operation seen the other one?” Conflict detection uses the second question; wall-clock time only helps choose a deterministic outcome.

Delivery / serverSeq

Total order at SuperSync

A per-user sequence supports pages, cursors, causal full-state boundaries, and bounded prefix retention. File providers have no server-assigned operation cursor. Their adapter exposes envelope syncVersion as a synthetic transport watermark/latestSeq for catch-up and gap detection. It is not the provider revision/ETag and does not prove per-operation ordering.

Knowledge / vector clock

Partial causal order

Each client counter records which histories an operation includes. Comparison yields before, after, equal, or concurrent.

Entity / frontier

Conflict comparison base

Retained operations plus the snapshot clock conservatively reconstruct the newest causal knowledge for an entity.

Device A / title edit

{ A: 2, B: 1 }

Device A has seen B's first operation, then changes the task title while offline.

Device B / completion edit

{ A: 1, B: 2 }

Device B has seen A's first operation, then marks the same task done while offline.

↯

Neither clock contains the other: the edits are concurrent

Because title and completion are disjoint eligible fields, the client can synthesize one partial update that preserves both. If the edits overlap or cannot be decomposed safely, the policy continues through explicit precedence and deterministic LWW.

Policy 01

Merge safe disjoint fields

Eligible single-entity updates with known, non-overlapping deltas produce a deterministic, byte-stable changes payload. Each resolving client still creates its own operation identity and clock.

Policy 02

Apply semantic precedence

Archive operations and marked project deletions follow their explicit delete/archive-wins rules instead of generic timestamps.

Policy 03

Fall back to deterministic LWW

The newest operation timestamp wins; an exact millisecond tie uses stable client-ID ordering so both devices choose identically.

Policy 04

Dominate and propagate

A local or merged winner becomes a new operation whose clock covers both histories, then re-uploads through the ordinary protocol.

!

Vector clocks detect conflicts; timestamps resolve some conflicts

Comparing timestamps first would silently order genuinely independent work. Likewise, pruning an incoming clock before server comparison can erase the edge needed to prove that a retry dominates the operation it replaces. The resolver heals supported no-pending crossings, but cases with compacted-away evidence or atomic operations it cannot decompose remain documented “Composition residual” cases.

Deep / conflict evidence and manual review

The device-local journal and review UI are observe-only capabilities: recording never affects the winner and is intentionally not atomic with the op log. The main production remote-resolution path currently disables journal emission, so the journal must not be treated as a complete audit trail. Where entries exist, “flip” dispatches a new ordinary edit when the discarded delta can be reapplied safely. See Conflict Journal, Disjoint-Field Auto-Merge & Review UI.

07 / facet: exceptional operations

Replace, repair, or abort—never blur them

Full-state operations and provider configuration changes cross wider causal boundaries than ordinary entity edits. They deliberately use different filtering and fencing rules rather than pretending to be large updates.

Explicit boundary

SYNC_IMPORT / BACKUP_IMPORT

Replace the synced projection as a clean slate while preserving device-local sync configuration. Causally older and genuinely concurrent operations that never observed the import are intentionally dropped.

Causal maintenance

REPAIR

Modern causal repairs can name the server prefix their snapshot includes; their narrower policy preserves or repositions concurrent suffix work rather than treating it as a user-requested reset. Legacy or unscoped repairs lack that marker and require special prefix positioning.

Compatibility boundary

Receiver migration

Operations migrate before conflict checks. An unknown newer schema blocks the batch at that operation and freezes cursor progress until an updated client can interpret it.

Configuration boundary

Provider epoch

Provider, account, folder, or encryption changes invalidate old in-flight work. Normal target changes install the new configuration/provider, then bump the epoch; destructive remote writes use the stronger block-and-drain sequence below. Epoch assertions make stale continuations abort benignly across awaits.

  1. 01 / block

    Stop new cycles

    Raise the synchronous gate before waiting on any existing work.

  2. 02 / bump

    Invalidate immediately

    Bump the epoch so stale continuations fail at their next fence.

  3. 03 / drain

    Drain all old work

    Wait for the main cycle and guarded side-channel cycles to settle.

  4. 04 / mutate

    Run destructive work

    Change encryption state, force-upload, or perform the other fenced remote mutation.

  5. 05 / retry

    Fresh cycle starts

    After unblocking, download from the new target before sending local work.

!

A schema bump is not a compatibility shield

Some released clients apply newer operations unmigrated; newer receivers correctly block but are then fenced until update. New semantics must degrade safely for the released fleet through payload markers or envelopes. Bump only when a transformation or hard semantic barrier genuinely requires it.

08 / close the loop

One restart tests every durability promise

Startup chooses the best trustworthy baseline, reconstructs its remaining tail, applies the path-specific trust or validation rule, and resumes interrupted remote checkpoints before a reconstructed projection may become a new baseline.

  1. 01 / open

    Initialize storage

    Open the adapter and inspect durable operation status.

  2. 02 / baseline

    Screen snapshot

    Check structure and schema compatibility before trusting state_cache.

  3. 03 / project

    Load live state

    Hydrate the snapshot or begin from the initial projection.

  4. 04 / tail

    Direct-load or migrate + replay

    Load the final operation directly when it carries full state and no pending reducer work remains. Otherwise migrate replayable operations and bulk-apply the result; migration may transform, split, or drop obsolete rows.

  5. 05 / recover

    Finish checkpoints

    Resume pending reducer or archive work without redoing completed effects.

  6. 06 / trust

    Gate the next baseline

    Trust a matching-schema cache on its fast path; validate directly loaded, migrated, or replayed results before saving a reconstructed replacement.

!

A snapshot is an accelerator, not permission to discard recovery data

If an intact snapshot is unreadable to this build and only a compacted tail survives, replay may produce a useful partial projection—but it must not overwrite the last complete snapshot or prune the remaining operations. Compaction stays disabled until a clean boot restores confidence.

Normal restart

Snapshot + tail

Load a structurally usable cache, reconstruct the retained tail with the guarded path above, finish checkpoints, then continue capture and sync.

Fallback restart

Replay without destroying evidence

With no usable cache, reconstruct from retained history when it is sufficient. A degraded fallback stays visibly degraded and cannot compact over better data.
